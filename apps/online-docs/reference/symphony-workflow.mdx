Symphony `WORKFLOW.md` has two parts:

- **YAML front-matter** for orchestrator config
- **Markdown body** for the fallback monolith prompt

When `prompts` front-matter is configured, runtime prompts come from those prompt files. Without per-state prompts, the markdown body after the second `---` is rendered as the fallback monolith prompt.

## Worker tracker contract (current)

In worker prompts/skills, use the backend-neutral helper script for tracker/document/state operations:

```bash
.agents/skills/sym-state/scripts/sym-call <operation> --input /tmp/input.json
```

Supported operations:

- `issue.get`
- `issue.list-children`
- `document.read`
- `document.write`
- `comment.upsert`
- `issue.update-state`
- `issue.create-followup`

Do **not** use legacy `kata_*` worker tools or backend-specific tracker mutation paths in normal worker flow.

## Agent harness vs backend-state separation

- `agent.*` config controls the **runtime harness** (`name`, `command`, `model`, timeouts, sandbox/approval behavior).
- Tracker/document/state actions are handled through the **sym-state helper contract** above.

## Minimal current example

```yaml
---
tracker:
  kind: linear
  api_key: $LINEAR_API_KEY
  project_slug: my-project
  active_states: [Todo, In Progress, Agent Review, Merging, Rework]
  terminal_states: [Closed, Cancelled, Canceled, Duplicate, Done]

workspace:
  root: ~/symphony-workspaces
  repo: https://github.com/example/project.git
  git_strategy: auto
  isolation: local
  branch_prefix: symphony
  clone_branch: main
  base_branch: main

agent:
  name: pi
  command: pi --mode rpc
  model: anthropic/claude-opus-4-6
  no_session: true
  max_concurrent_agents: 1
  max_turns: 20
  approval_policy: never
  thread_sandbox: danger-full-access
---
```

## Key fields

### `tracker`

- `kind`: `linear` or `github`
- `api_key`: token (`$VAR` supported)
- Linear: `project_slug`, optional `workspace_slug`
- GitHub: `repo_owner`, `repo_name`, optional `github_project_number`
- `active_states`, `terminal_states`, optional `exclude_labels`

> GitHub label-mode guidance is legacy and not documented as the active workflow path.

### `workspace`

- `root`, `repo`, `base_branch`, `branch_prefix`
- `git_strategy`:
  - `auto` (default)
  - `clone-local`
  - `clone-remote`
  - `worktree` (local-path repo only)

Worktree bootstrap notes:

- requires `workspace.repo` to be a local path
- uses `git worktree add`
- cleanup uses `git worktree remove`

### `agent`

- runner selection: `name` (`pi` or `codex`)
- process launch: `command`
- model controls: `model`, optional `model_by_state`
- runtime controls: `max_concurrent_agents`, `max_turns`, `stall_timeout_ms`, `read_timeout_ms`
- safety/sandbox: `approval_policy`, `thread_sandbox`, `turn_sandbox_policy`

### `prompts`

Optional per-state prompt files:

```yaml
prompts:
  system: prompts/system.md
  repo: prompts/repo.md
  by_state:
    Todo: prompts/in-progress.md
    In Progress: prompts/in-progress.md
    Agent Review: prompts/agent-review.md
    Merging: prompts/merging.md
    Rework: prompts/rework.md
  default: prompts/in-progress.md
```

Prompt concatenation order: `system` → `repo` → `shared` (legacy) → state prompt.
