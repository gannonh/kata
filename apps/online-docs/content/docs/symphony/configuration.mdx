# Configuration

All Symphony configuration lives in the YAML front-matter of a `WORKFLOW.md` file. Everything below the `---` is the prompt template (a Liquid template rendered per agent session).

## GUI Editor (`/symphony config`)

Kata CLI includes a guided editor for WORKFLOW frontmatter:

- Run `/symphony config` to edit the default workflow path
- Run `/symphony config <path>` to edit a specific file
- Path resolution order: explicit path argument → `symphony.workflow_path` preference → `./WORKFLOW.md`

The editor groups fields by section (`tracker`, `workspace`, `agent`, `kata_agent`, `notifications`, `prompts`, `server`, `hooks`, `worker`) and uses type-aware controls:

- Text input for strings
- Number input for integer fields
- Enum pickers for values like `backend`, `git_strategy`, `isolation`, `codex_auth`
- Boolean toggles for flags like `cleanup_on_done` and `no_session`
- Multi-line string-array editing for fields like `events`, `ssh_hosts`, and Docker lists

Before saving, validation checks required fields, enum values, numeric ranges, and common invalid combinations (for example `worktree` + remote repo, or `docker` + `clone-local`).

Save writes only the frontmatter block, preserving the markdown prompt body and existing YAML comments.

## Minimal Example

```yaml
---
tracker:
  kind: linear
  api_key: $LINEAR_API_KEY
  project_slug: my-project

workspace:
  root: ~/symphony-workspaces
  repo: https://github.com/you/repo.git

agent:
  backend: kata-cli

kata_agent:
  command: kata
  model: anthropic/claude-sonnet-4-6
---

You are working on {{ issue.identifier }}: {{ issue.title }}.
{{ issue.description }}
```

## Config Sections

### `tracker`

Linear connection and issue filtering.

| Field | Type | Default | Description |
|-------|------|---------|-------------|
| `kind` | string | *(required)* | Must be `"linear"` |
| `api_key` | string | *(required)* | Linear personal API key. Supports `$VAR` env indirection |
| `project_slug` | string | *(required)* | Linear project URL slug |
| `workspace_slug` | string | `"kata-sh"` | Linear workspace slug for dashboard links |
| `assignee` | string | *(none)* | Filter to this Linear username |
| `active_states` | string[] | `["Todo", "In Progress"]` | States eligible for dispatch |
| `terminal_states` | string[] | `["Closed", "Cancelled", "Canceled", "Duplicate", "Done"]` | States that mark work complete |

### `polling`

| Field | Type | Default | Description |
|-------|------|---------|-------------|
| `interval_ms` | u64 | `30000` | Poll interval in milliseconds |

### `workspace`

Where and how agent workspaces are created.

| Field | Type | Default | Description |
|-------|------|---------|-------------|
| `root` | string | `$TMPDIR/symphony_workspaces` | Root directory for workspaces. Supports `~` |
| `repo` | string | *(none)* | Repository URL or local path |
| `git_strategy` | string | `"auto"` | `clone-local`, `clone-remote`, `worktree`, or `auto` |
| `isolation` | string | `"local"` | `local` or `docker` |
| `branch_prefix` | string | `"symphony"` | Prefix for auto-created branches |
| `clone_branch` | string | *(none)* | Source branch for clone-based bootstraps |
| `base_branch` | string | `"main"` | Base branch for merge targets |
| `cleanup_on_done` | bool | `false` | Remove workspace when issue reaches terminal state |

### `agent`

| Field | Type | Default | Description |
|-------|------|---------|-------------|
| `max_concurrent_agents` | u32 | `10` | Global cap on simultaneous agent sessions |
| `max_turns` | u32 | `20` | Maximum prompt turns per session |
| `max_retry_backoff_ms` | u64 | `300000` | Maximum exponential backoff between retries |
| `backend` | string | `"codex"` | `codex` or `kata-cli` (alias: `kata`) |

### `kata_agent` (alias: `pi_agent`)

Config for the Kata CLI backend.

| Field | Type | Default | Description |
|-------|------|---------|-------------|
| `command` | string/string[] | `["kata"]` | Kata CLI executable and base args |
| `model` | string | *(none)* | Default model override |
| `model_by_state` | map | `{}` | Per-state model overrides (keys are Linear state names) |
| `no_session` | bool | `true` | Pass `--no-session` to disable persistent sessions |
| `append_system_prompt` | string | *(none)* | Path passed via `--append-system-prompt` |
| `read_timeout_ms` | u64 | `5000` | Timeout waiting for process output |
| `stall_timeout_ms` | u64 | `300000` | Time before non-progressing session is stalled |

### `codex`

Config for the Codex backend.

| Field | Type | Default | Description |
|-------|------|---------|-------------|
| `command` | string/string[] | `["codex", "app-server"]` | Codex executable and arguments |
| `approval_policy` | object | *(reject all)* | Codex approval policy |
| `thread_sandbox` | string | `"workspace-write"` | Codex sandbox mode |
| `turn_timeout_ms` | u64 | `3600000` | Hard timeout per turn (1 hour) |
| `stall_timeout_ms` | u64 | `300000` | Time before non-progressing session is stalled |

### `notifications`

Outbound webhook notifications for events needing human attention.

| Field | Type | Default | Description |
|-------|------|---------|-------------|
| `slack.webhook_url` | string | *(none)* | Slack incoming webhook URL. Supports `$VAR` |
| `slack.events` | string[] | `[]` | State transitions: `todo`, `in_progress`, `agent_review`, `human_review`, `merging`, `rework`, `done`, `closed`, `cancelled`. Runtime: `stalled`, `failed`. Wildcard: `all` |

When omitted, notifications are disabled. See [Notifications](/docs/symphony/notifications) for details.

### `hooks`

Shell commands at workspace lifecycle points.

| Field | Type | Default | Description |
|-------|------|---------|-------------|
| `after_create` | string | *(none)* | After workspace created |
| `before_run` | string | *(none)* | Before agent session starts |
| `after_run` | string | *(none)* | After agent session ends |
| `before_remove` | string | *(none)* | Before workspace removed |
| `timeout_ms` | u64 | `60000` | Timeout for each hook |

Hooks receive: `SYMPHONY_ISSUE_ID`, `SYMPHONY_ISSUE_IDENTIFIER`, `SYMPHONY_ISSUE_TITLE`, `SYMPHONY_WORKSPACE_PATH`.

### `server`

| Field | Type | Default | Description |
|-------|------|---------|-------------|
| `port` | u16 | *(none)* | HTTP dashboard and API port |
| `host` | string | `"127.0.0.1"` | HTTP bind address |

### `worker` (SSH)

| Field | Type | Default | Description |
|-------|------|---------|-------------|
| `ssh_hosts` | string[] | `[]` | Remote SSH hosts for distributed sessions |
| `max_concurrent_agents_per_host` | u32 | *(none)* | Per-host concurrency cap |

## Environment Variable Indirection

Any string value starting with `$` followed by a bare identifier is resolved from the environment at startup:

```yaml
tracker:
  api_key: $LINEAR_API_KEY
```

## Prompt Template Variables

The markdown body below `---` is a Liquid template with these variables:

| Variable | Description |
|----------|-------------|
| `{{ issue.identifier }}` | e.g. `KAT-42` |
| `{{ issue.title }}` | Issue title |
| `{{ issue.state }}` | Current Linear state |
| `{{ issue.description }}` | Issue body markdown |
| `{{ issue.labels }}` | Comma-separated label names |
| `{{ issue.url }}` | Linear issue URL |
| `{{ issue.children_count }}` | Number of child issues |
| `{{ issue.parent_identifier }}` | Parent issue identifier (if sub-issue) |
| `{{ workspace.base_branch }}` | Configured base branch |
| `{{ attempt }}` | Current retry attempt number |
