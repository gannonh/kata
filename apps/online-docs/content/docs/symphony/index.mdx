# Symphony

Symphony is a headless orchestrator that polls a Linear project for issues and dispatches parallel agent sessions to work on them. You run Symphony, point it at a Linear project, and it picks up tickets, clones your repo, runs an agent on each issue, creates PRs, addresses review feedback, and merges — all without human intervention.

## How It Works

1. Symphony polls your Linear project for issues in active states (e.g. `Todo`, `In Progress`)
2. For each issue, it creates an isolated workspace (clone or worktree of your repo) and starts an agent session
3. The agent reads the issue, writes code, runs tests, creates a PR, and handles review feedback
4. When the issue reaches a terminal state (`Done`, `Closed`), the workspace is cleaned up
5. Multiple issues run in parallel, up to your configured concurrency limit

All configuration — tracker, workspace, agent, notifications, and the prompt template — lives in a single `WORKFLOW.md` file.

## Agent Backends

Symphony supports two agent backends:

| Backend | Config value | What it runs | Models |
|---------|-------------|--------------|--------|
| **Kata CLI** | `kata-cli` (alias: `kata`) | Kata CLI in RPC mode | Any model supported by pi-ai: Anthropic, OpenAI, Google, Mistral, Bedrock, Azure |
| **Codex** | `codex` | OpenAI Codex app-server | OpenAI Codex models |

## Installation

### Pre-built binaries

Download from [GitHub Releases](https://github.com/gannonh/kata/releases):

| Platform | Binary |
|----------|--------|
| macOS (Apple Silicon) | `symphony-macos-arm64` |
| Linux (x86_64) | `symphony-linux-x86_64` |
| Windows (x86_64) | `symphony-windows-x86_64.exe` |

### Build from source

Requires [Rust toolchain](https://rustup.rs/):

```bash
git clone https://github.com/gannonh/kata.git
cd kata/apps/symphony
cargo build --release
# Binary at: target/release/symphony
```

## Quick Start

```bash
# 1. Set up environment
cp .env.example .env
# Edit .env with LINEAR_API_KEY and agent auth

# 2. Create a WORKFLOW.md (see Configuration)

# 3. Run
symphony WORKFLOW.md --port 8080
```

Symphony starts polling Linear. Open `http://localhost:8080` for the web dashboard, or watch the built-in terminal dashboard (enabled by default).

## Kata CLI Integration

Kata CLI ships a built-in Symphony client extension. When `symphony.url` is configured in [preferences](/docs/cli/preferences), the following are available:

### Slash Commands

| Command | Description |
|---------|-------------|
| `/symphony status` | Fetch live worker/queue state from `GET /api/v1/state` |
| `/symphony watch KAT-920` | Stream live events for an issue from `GET /api/v1/events` |
| `/symphony watch KAT-920 --max-events 10` | Stop after 10 events |
| `/symphony watch KAT-920 --timeout-ms 60000` | Watch for up to 60 seconds |
| `/symphony console` | Toggle the live in-chat operator dashboard panel |
| `/symphony console off` | Close the panel and stop live stream updates |
| `/symphony console refresh` | Force a manual snapshot refresh (`Ctrl+Alt+S`) |

### Agent Tools

The extension also exposes tools the agent can call programmatically:

| Tool | Description |
|------|-------------|
| `symphony_status` | Returns orchestrator state as structured JSON |
| `symphony_watch` | Streams issue events, returns structured array |
| `symphony_logs` | Placeholder — not yet available |
| `symphony_steer` | Placeholder — not yet available |

### Configuration

Set Symphony preferences in `.kata/preferences.md`:

```yaml
symphony:
  url: http://localhost:8080
  console_position: below-output # or above-status
```

Or via environment variables for URL fallback: `KATA_SYMPHONY_URL` or `SYMPHONY_URL`.

When the console panel shows pending escalations, respond directly in chat input:

- `!respond <answer>` (auto-target if exactly one escalation is pending)
- `!respond <request-id|index> <answer>` (required for multiple pending escalations)

## Ticket Lifecycle

```
Todo → In Progress → Agent Review → Human Review → Merging → Done
                         ↑               |
                         └── Rework ←────┘
```

| Status | Who sets it | What happens |
|--------|------------|--------------|
| **Todo** | Human | Issue is queued — Symphony picks it up on the next poll |
| **In Progress** | Orchestrator | Agent is working — writing code, running tests |
| **Agent Review** | Agent | Agent addresses PR review comments |
| **Human Review** | Agent | PR is ready for human approval |
| **Merging** | Human | Agent merges the approved PR |
| **Rework** | Human | Agent scraps current approach, starts fresh |
| **Done** | Agent | Terminal — PR merged, workspace cleaned up |
