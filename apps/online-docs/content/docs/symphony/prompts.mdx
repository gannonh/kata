# Per-State Prompts

Instead of sending one monolithic prompt for every agent session, Symphony can select a focused prompt based on the issue's current Linear state. An agent implementing code gets implementation instructions; an agent addressing PR feedback gets review instructions.

## Configuration

Add a `prompts` section to your WORKFLOW.md front-matter:

```yaml
prompts:
  shared: prompts/shared.md           # prepended to every prompt
  by_state:
    Todo: prompts/in-progress.md
    In Progress: prompts/in-progress.md
    Agent Review: prompts/agent-review.md
    Merging: prompts/merging.md
    Rework: prompts/rework.md
  default: prompts/in-progress.md      # fallback for unlisted states
```

Paths are resolved relative to the WORKFLOW.md file.

## How It Works

1. Symphony dispatches an issue and checks its current Linear state
2. It looks up the state name in `prompts.by_state` (case-insensitive)
3. If found, it reads that file and prepends the `shared` file content
4. If not found, it falls back to `prompts.default`
5. The combined prompt is rendered as a Liquid template with the standard variables

## Prompt Files

| File | When used | Typical job |
|------|-----------|-------------|
| `shared.md` | Every dispatch | Repo context, tool instructions, workpad protocol |
| `in-progress.md` | `Todo`, `In Progress` | Implement, test, push, open PR, move to Agent Review |
| `agent-review.md` | `Agent Review` | Address PR review comments, move to Human Review |
| `merging.md` | `Merging` | Merge the approved PR, move to Done |
| `rework.md` | `Rework` | Close existing PR, start fresh on a new branch |

## Issue Shape Detection

The `in-progress.md` prompt can detect issue shape using template variables:

```liquid
{% if issue.children_count > 0 %}
  {# Kata-planned slice — parent issue with child tasks #}
{% elsif issue.parent_identifier %}
  {# Individual task — child of a slice #}
{% else %}
  {# Flat ticket — standalone issue #}
{% endif %}
```

This means one workflow file handles both flat tickets and hierarchical Kata-planned slices.

## Backward Compatibility

Without a `prompts` section, Symphony uses the markdown body after `---` as the prompt for all states — identical to pre-1.2.0 behavior.

## Session State Changes

When an agent moves an issue to a different state mid-session (e.g. `In Progress` → `Agent Review`), the multi-turn loop detects the state change and ends the session. The orchestrator then re-dispatches the issue with the correct prompt for the new state.
