Kata CLI (`@kata-sh/cli`) provides backend IO and setup commands used by Kata skills. It manages setup, backend validation, and operation dispatch.

## Quick start

Run setup in a repository checkout:

```bash
npx @kata-sh/cli setup --yes --repo owner/repo --project-number 12
npx @kata-sh/cli doctor
```

Or install globally:

```bash
npm install -g @kata-sh/cli
kata setup --yes --repo owner/repo --project-number 12
kata doctor
```

## Core command surface

- `kata setup` - install skills and create `.kata/preferences.md`
- `kata doctor` - validate harness install, backend config, auth, and project fields
- `kata call <operation> --input <request.json>` - run a single backend operation
- `kata json <request.json>` - run a backend operation from a JSON envelope

See [Commands](/cli/commands) for details.

## Backend behavior

Preferences live at `.kata/preferences.md`.

`workflow.mode` supports:

- `github` - GitHub Projects v2 backend
- `linear` - Linear backend

Setup provisions GitHub mode by default and Linear mode when `--backend linear` is selected.

GitHub setup writes:

```yaml
---
workflow:
  mode: github
github:
  repoOwner: kata-sh
  repoName: kata-mono
  stateMode: projects_v2
  githubProjectNumber: 12
---
```

Linear setup writes `linear.workspace`, `linear.team`, `linear.project`, `linear.authEnv`, and the default state mapping. See [Linear integration](/cli/linear-integration) and [CLI preferences](/reference/cli-preferences) for field reference.

## Skill-helper workflow

Installed Kata skills call Kata CLI via helper scripts:

```bash
node ./scripts/kata-call.mjs setup
node ./scripts/kata-call.mjs doctor
node ./scripts/kata-call.mjs issue.get --input /tmp/kata-issue-get.json
```

This keeps workflow logic backend-neutral while using the same backend operation contract.
