---
title: "Agent runners"
description: "Configure Symphony worker runtime runners (Pi or Codex)."
---

Symphony uses `agent.*` to configure the worker runtime harness.

| Runner | `agent.name` | Typical `agent.command` |
| --- | --- | --- |
| Pi | `pi` | `pi --mode rpc` |
| Codex | `codex` | `codex app-server` |

<Note>
  This runtime selection is separate from tracker, document, and state operations. In worker prompts, call `"$SYMPHONY_BIN" helper <operation> --workflow "$SYMPHONY_WORKFLOW_PATH" --input "$INPUT"` for backend-neutral actions.
</Note>

## Pi runner (default in Kata workflows)

```yaml
agent:
  name: pi
  command: pi --mode rpc
  model: anthropic/claude-opus-4-6
  model_by_state:
    Agent Review: anthropic/claude-sonnet-4-6
    Merging: anthropic/claude-sonnet-4-6
  no_session: true
  max_concurrent_agents: 1
  max_turns: 20
  approval_policy: never
  thread_sandbox: danger-full-access
  stall_timeout_ms: 300000
  read_timeout_ms: 5000
```

## Codex runner

```yaml
agent:
  name: codex
  command: codex app-server
  model: gpt-5-codex
  max_concurrent_agents: 1
  max_turns: 20
  approval_policy: never
  thread_sandbox: danger-full-access
  turn_timeout_ms: 3600000
  stall_timeout_ms: 300000
  read_timeout_ms: 5000
```

## Key fields

- `agent.name`: runner protocol (`pi` or `codex`)
- `agent.command`: executable + fixed args
- `agent.model`: default model override
- `agent.model_by_state`: per-workflow-state model overrides
- `agent.no_session`: Pi-only `--no-session` behavior
- `agent.max_concurrent_agents`, `agent.max_turns`: execution limits
- `agent.approval_policy`, `agent.thread_sandbox`, `agent.turn_sandbox_policy`: safety controls
- `agent.stall_timeout_ms`, `agent.read_timeout_ms`, `agent.turn_timeout_ms`: timeout controls

For complete config details, see `/reference/symphony-workflow`.
