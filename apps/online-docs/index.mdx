## Quick start

<Steps>
  <Step title="Run setup">
    Bootstrap Kata CLI with `npx`:
    ```bash
    npx @kata-sh/cli setup --yes --backend github --repo owner/repo --project-number 12
    ```
  </Step>
  <Step title="Validate your environment">
    Run doctor to verify backend config, auth, and required project fields:
    ```bash
    npx @kata-sh/cli doctor
    ```
  </Step>
  <Step title="Run backend operations">
    Use `npx @kata-sh/cli call <operation> --input <request.json>` and `npx @kata-sh/cli json <request.json>` to execute backend operations from scripts and skills. If you installed globally, run the same commands with `kata`.
  </Step>
  <Step title="Open Desktop or dispatch with Symphony">
    Use Kata Desktop for a visual Pi session, or run `symphony` from a project with `.symphony/WORKFLOW.md` to manage tracker-driven agent sessions.
  </Step>
</Steps>

## What makes Kata different

<AccordionGroup>
  <Accordion title="Backend-neutral workflow operations">
    Kata skills call Kata CLI for issue, document, and state operations across supported tracker backends.
  </Accordion>
  <Accordion title="Pi-powered interactive sessions">
    Kata Desktop uses Pi in RPC mode for chat, tools, provider auth, and model execution.
  </Accordion>
  <Accordion title="Tracker-integrated execution">
    Kata Symphony polls tracker issues, dispatches agents in parallel, and moves work through configured workflow states.
  </Accordion>
  <Accordion title="Full PR lifecycle management">
    Kata creates branches, opens PRs, runs code review, addresses feedback, and merges autonomously or with approval gates you control.
  </Accordion>
  <Accordion title="Codebase context tooling">
    Kata Context indexes repositories with structural and semantic signals so agents can query symbols, dependencies, and related files.
  </Accordion>
</AccordionGroup>
