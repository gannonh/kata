Kata Skill System

Three layers of skill management — pi framework discovery, Kata preference policy, and auto-mode hot discovery — all converging in the system prompt

Overview — Three Mechanisms
The big picture

Skills are SKILL.md files with YAML frontmatter (name + description) and markdown instructions. They are never loaded into context automatically — only their names, descriptions, and file paths are listed in the system prompt. The agent uses read to load a skill's content on demand when the task matches.

MechanismOwnerWhat it doesWhen it fires
<available_skills> pi framework Discovers all skills from disk, injects name + description + path into system prompt Every session, at system prompt build time
Kata Skill Preferences Kata extension Adds policy layer: "always use X, prefer Y, avoid Z" with resolved paths Every session, in before_agent_start hook
Newly Discovered Skills Kata extension Detects skills installed mid-run, injects them into subsequent sessions Auto-mode only, rare (only when agent installs a skill during execution)
Key principle: Skill file contents are never injected into the system prompt. Only paths and descriptions are listed. The agent reads skill files on demand via the read tool. This keeps the system prompt lean — skill files can be 1000+ lines and there can be dozens of them.
Skill Sources — Where Skills Live
Filesystem locations scanned by pi

Pi scans these in order. First skill found for a given name wins. Symlink dedup prevents the same file from loading twice.

Skill file structure

A valid skill needs a directory with a SKILL.md file containing YAML frontmatter:

my-skill/ ├── SKILL.md ← Required: frontmatter (name + description) + instructions ├── scripts/ ← Optional: helper scripts ├── references/ ← Optional: detailed docs loaded on demand └── commands/ ← Optional: slash command templates

The name field must match the parent directory. The description field determines when the agent selects the skill — it's the only thing visible in the system prompt.

Pi Framework Discovery — loadSkills()
system-prompt.ts · skills.ts

At startup, pi calls loadSkills() which scans all skill directories, deduplicates by name (first wins) and by realpath (symlinks), validates frontmatter, and returns a Skill[] array.

Then buildSystemPrompt() calls formatSkillsForPrompt(skills) which produces the XML block:

<available_skills> <skill> <name>debug-like-expert</name> <description>Deep analysis debugging mode for complex issues...</description> <location>/Users/you/.kata-cli/agent/skills/debug-like-expert/SKILL.md</location> </skill> <skill> <name>frontend-design</name> <description>Create distinctive, production-grade frontend interfaces...</description> <location>/Users/you/.kata-cli/agent/skills/frontend-design/SKILL.md</location> </skill> <!-- ... every skill from all scanned directories ... --> </available_skills>

This XML block is appended to pi's base system prompt before Kata's before_agent_start hook fires. It contains every skill from every source — bundled, user-installed, project-local, and cross-harness.

Gating: Pi only injects the <available_skills> block if the read tool is available. Without read, the agent can't load skill files, so listing them would be useless noise.
Kata Bundled Skills — Sync on Launch
resource-loader.ts

On every Kata launch, the resource loader copies bundled skills from the CLI package into the user's agent directory:

Source (in CLI package)Destination
src/resources/skills/debug-like-expert/~/.kata-cli/agent/skills/debug-like-expert/
src/resources/skills/frontend-design/~/.kata-cli/agent/skills/frontend-design/
src/resources/skills/swiftui/~/.kata-cli/agent/skills/swiftui/

Uses cpSync(src, dest, { recursive: true, force: true }) — always overwrites so skill updates land on next launch.

These skills end up in the same directory pi scans, so they appear in <available_skills> automatically. Kata also hardcodes them in the system.md skills table as a redundant hint.

system.md Skills table (redundant hint)

The system.md prompt template contains a hardcoded table listing the 3 bundled skills with trigger descriptions. This is a redundant reinforcement — the same skills are already in <available_skills> from pi's discovery. The table provides more specific trigger guidance than the generic description field.

Kata Skill Preferences — Policy Layer
preferences.ts · .kata/preferences.md

Users can configure skill selection policy in .kata/preferences.md (project) and/or ~/.kata-cli/agent/preferences.md (global). This adds a priority layer on top of pi's discovery — it doesn't replace it.

Preference fields

FieldEffect
always_use_skillsLoad and follow when the task matches — don't treat as a vague suggestion
prefer_skillsPrefer over other skills when relevant
avoid_skillsSkip unless clearly needed
skill_rulesSituational: when: "releasing" → use: [releasing-kata]
custom_instructionsFreeform instructions appended to the block

Path resolution

resolveAllSkillReferences() resolves each skill reference at session start:

  • Absolute path — checked directly (/Volumes/.../SKILL.md)
  • Absolute directory — looks for SKILL.md inside (/Volumes/.../releasing-kata/)
  • Tilde path — expanded from ~/ (~/.agents/skills/pull-requests/SKILL.md)
  • Bare name — scanned in all known skill directories (pull-requests)
  • Unresolved references produce a warning in the system prompt

What gets injected

renderPreferencesForSystemPrompt() produces:

## Kata Skill Preferences - Treat these as explicit skill-selection policy for Kata work. - If a listed skill exists and is relevant, load and follow it instead of treating it as a vague suggestion. - Current user instructions still override these defaults. - Always use these skills when relevant: - /Volumes/EVO/kata/kata-mono/.agents/skills/releasing-kata/SKILL.md - /Users/gannonhall/.agents/skills/pull-requests/SKILL.md

Again — only paths are injected, not content. The agent uses read on the path when it decides the task matches.

Auto-mode Hot Discovery
skill-discovery.ts · auto.ts

Handles the rare case where the agent itself installs a new skill during auto-mode execution (e.g., via pi install). Subsequent auto-mode sessions need to see the new skill without a full restart.

Lifecycle

EventActionCode
/kata auto startsSnapshot current skill directory listingsnapshotSkills()
Each new auto-mode sessionDiff current listing against snapshotdetectNewSkills()
New skill foundParse its SKILL.md frontmatterparseSkillFrontmatter()
Inject into system promptAppend XML blockformatSkillsXml()
/kata stopClear snapshotclearSkillSnapshot()

If new skills are detected, the following is appended to the system prompt:

<newly_discovered_skills> The following skills were installed during this auto-mode session. Use the read tool to load a skill's file when the task matches its description. <skill> <name>my-new-skill</name> <description>Does something useful</description> <location>/Users/you/.kata-cli/agent/skills/my-new-skill/SKILL.md</location> </skill> </newly_discovered_skills>

Controlled by the skill_discovery preference (auto | off). When off, the snapshot is never taken and this mechanism is disabled.

Complete system prompt structure

All three skill mechanisms converge in a single system prompt. Here's the exact order:

┌─ Pi base system prompt ─────────────────────────────────────┐ │ Tool instructions, model guidelines, pi documentation │ │ │ │ <available_skills> pi framework │ │ Every skill from all directories │ │ (name + description + path only) │ │ </available_skills> │ └───────────────────────────────────────────────────────────────┘ ┌─ [SYSTEM CONTEXT — Kata] ───────────────────────────────────┐ │ │ │ system.md Kata extension │ │ Agent identity, hard rules, skills table (redundant hint), │ │ commands, execution heuristics, playbooks │ │ │ │ ## Workflow Mode dynamic string │ │ "mode: linear — Follow Linear mode instructions..." │ │ │ │ ## Kata Skill Preferences preferences.md │ │ always_use: [path1, path2] │ │ prefer: [path3] │ │ avoid: [path4] │ │ skill_rules: [{when: ..., use: ...}] │ │ │ │ <newly_discovered_skills> auto-mode only │ │ Skills installed during current auto-mode run (if any) │ │ </newly_discovered_skills> │ │ │ │ KATA-WORKFLOW.md workflow protocol │ │ ~550 lines: hierarchy, artifacts, phases, formats, etc. │ │ │ └───────────────────────────────────────────────────────────────┘
Runtime Behavior — How the Agent Uses Skills
Agent decision loop

The agent sees all three skill blocks in its system prompt. When a user request arrives, it:

  1. Scans <available_skills> — matches the task against skill descriptions
  2. Checks Kata Skill Preferences — if a matching skill is in always_use or prefer, it takes priority; if in avoid, it's deprioritized
  3. Calls read on the skill's <location> path to load the full SKILL.md content
  4. Follows the skill's instructions to execute the task

If a skill references relative paths (e.g., ./references/api.md), the agent resolves them against the skill directory. The skill is a self-contained capability package — instructions, scripts, references, and templates all live in its directory.

What does NOT happen
  • Skills are never auto-loaded — the agent decides when to read a skill based on task match
  • Skill content is never in the system prompt — only names, descriptions, and paths
  • Preferences don't override pi's discovery — they add policy on top of it
  • New skill discovery is not general hot-reload — it only catches skills added to ~/.kata-cli/agent/skills/ during the current auto-mode session, not edits to existing skills
