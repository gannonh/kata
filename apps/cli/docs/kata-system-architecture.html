Kata System Architecture

How the harness boots, injects context, dispatches work, and loops through tasks.
A visual reference for understanding the Kata CLI internals.

1 — Big Picture

Kata is a thin branded wrapper around pi-coding-agent. It sets env vars, syncs bundled resources, and launches a pi session with custom extensions that orchestrate a structured planning methodology.

Entry Point
loader.ts
Sets 10+ KATA_* env vars, pushes --mcp-config into argv, then dynamically imports cli.ts
Session Bootstrap
cli.ts
Calls initResources(), seeds MCP package, creates agent session via createAgentSession(), starts interactive mode
Resource Sync
resource-loader.ts
Copies bundled extensions/, agents/, skills/, and AGENTS.md to ~/.kata-cli/agent/ on every launch
Key principle: Kata never forks pi. It's a branded app built on top of the pi SDK, using env vars and extensions to customize behavior. All intelligence lives in extensions that register commands, hooks, tools, and UI components.

3 — Context Injection (What the LLM Sees)

The most critical part of the harness. The before_agent_start hook assembles a layered system prompt and optionally injects a hidden context message. This determines everything the LLM knows before it starts working.

System Prompt Assembly
The before_agent_start hook in kata/index.ts fires before every agent turn. It checks for a .kata/ directory and, if present, builds a layered system prompt injection:
{pi's base system prompt}

[SYSTEM CONTEXT — Kata]

## Kata — Kata Workflow              ← prompts/system.md (hard rules, heuristics, playbooks)
## Workflow Mode                     ← Linear mode notice (if applicable)
## Kata Skill Preferences            ← Resolved skill paths from preferences.md
## New Skills                        ← Skills installed during auto-mode session

{KATA-WORKFLOW.md}                   ← Full methodology doc (phases, artifacts, templates)
Layer 1: System Contract
prompts/system.md
Hard rules, execution heuristics, tool routing, verification requirements, communication style. Always injected into the system prompt when .kata/ exists.
Layer 2: Methodology
KATA-WORKFLOW.md
The full planning methodology: phases, artifact formats, git strategy, continue-here protocol, summary injection rules. Read directly from bundled path via KATA_WORKFLOW_PATH.
Layer 3: Preferences
preferences.md
User skill preferences resolved to absolute paths. Global (~/.kata-cli/) merged with project-local (.kata/). Rendered as a structured block in the system prompt.
Layer 4: Context Message
Guided Execute Context
For task execution: a hidden display:false message with inlined task plan, slice plan excerpt, carry-forward summaries from prior tasks, and resume state. Built by buildGuidedExecuteContextInjection().
Two injection channels: The system prompt return value modifies the LLM's persistent instructions. The optional message return value injects a hidden user-role message with task-specific context. Both are invisible in the chat UI (display: false).
4 — Commands & Entry Points

Two separate commands serve as entry points. They both ultimately dispatch the workflow protocol to the LLM, but with very different levels of intelligence.

/kata (Smart Wizard)
Contextual State Machine
Reads project state, shows TUI menus, bootstraps .kata/ if needed, detects crash recovery, and dispatches structured prompt templates with variable substitution.
  • /kata — smart entry wizard
  • /kata auto — auto-mode loop
  • /kata stop — stop auto-mode
  • /kata status — dashboard overlay
  • /kata queue — queue milestones
  • /kata discuss — discuss gray areas
  • /kata prefs — manage preferences
  • /kata doctor — diagnose/fix state
/kata-run (Raw Injection)
Protocol Dump
Reads KATA-WORKFLOW.md from disk, sends the full text as a hidden message with triggerTurn: true. No state awareness, no menus, no bootstrapping. The "escape hatch."
  • Reads KATA_WORKFLOW_PATH env var
  • Falls back to ~/.kata-cli/KATA-WORKFLOW.md
  • Optional user note appended as ## User Note
  • Agent follows protocol's "Fresh Session Checklist"
  • Undocumented

dispatchWorkflow() is the central dispatcher. Both /kata wizard and auto-mode route through it. It: (1) checks the workflow mode gate, (2) reads the protocol file (KATA-WORKFLOW.md or LINEAR-WORKFLOW.md), (3) combines it with a structured prompt, (4) sends a hidden message that triggers the LLM.

6 — Auto-Mode Loop

Auto-mode (/kata auto) is a state machine that creates fresh sessions for each unit of work. After each agent turn completes, it reads disk state, determines the next action, wipes context, and starts a new session with a focused prompt.

Loop Mechanics
  • agent_end hook fires after each turn
  • handleAgentEnd() saves activity log
  • Reads disk state via deriveState()
  • Determines next unit type from state
  • Creates fresh session via ctx.newSession()
  • Dispatches focused prompt for the next unit
  • Loop continues until milestone complete
Safety Mechanisms
  • Lock file on disk prevents concurrent auto-mode
  • Crash lock detected on next /kata invocation
  • Activity logs saved to .kata/activity/ as JSONL
  • Session forensics for crash recovery prompts
  • User Escape → pauses (not kills) auto-mode
  • Compaction blocked during auto-mode
  • session_shutdown saves state on Ctrl+C
7 — Hooks & Lifecycle Events

Kata registers four pi hooks that control the agent lifecycle. They're the nervous system connecting the wizard UI to the LLM's execution.

Hook When What It Does
before_agent_start Before every LLM turn Injects Kata system contract + methodology into system prompt. Optionally injects task plan, carry-forward context, and resume state as a hidden message. Only fires if .kata/ exists.
agent_end After each LLM turn completes Two jobs: (1) Check if discuss phase just finished → offer auto-start. (2) If auto-mode is active → save activity log, derive next state, advance to next unit via fresh session.
session_before_compact When pi is about to compact context Blocks compaction during auto-mode (each unit is already a fresh session). During manual mode, saves continue.md if mid-task so work can resume.
session_shutdown On Ctrl+C / SIGTERM Saves the current session's activity log. Leaves the lock file on disk so the next /kata can detect the interruption and offer recovery.

9 — Prompt Templates

The prompts/ directory contains 20+ Markdown templates with {{variable}} placeholders. loadPrompt() reads a template, validates all declared variables have values, and substitutes them.

guided-execute-task.md wizard auto milestoneId, sliceId, taskId, taskTitle Execute a specific task
guided-plan-slice.md wizard auto milestoneId, sliceId, sliceTitle Decompose slice into tasks
guided-complete-slice.md wizard auto milestoneId, sliceId, sliceTitle Write summary, UAT, squash-merge
guided-resume-task.md wizard milestoneId, sliceId Resume from continue.md
guided-research-slice.md wizard milestoneId, sliceId, sliceTitle Scout codebase before planning

How loadPrompt() Works 
// prompt-loader.ts
const promptsDir = join(dirname(import.meta.url), "prompts");

export function loadPrompt(name: string, vars: Record<string, string> = {}): string {
  const path = join(promptsDir, `${name}.md`);
  let content = readFileSync(path, "utf-8");

  // Validate: find all {{varName}} before substitution
  // (checking after would false-positive on {{...}} in inlined content)
  const declared = content.match(/\{\{[a-zA-Z][a-zA-Z0-9_]*\}\}/g);
  if (declared) {
    const missing = [...new Set(declared)].map(m => m.slice(2, -2))
      .filter(key => !(key in vars));
    if (missing.length > 0) throw new Error(`Missing vars: ${missing}`);
  }

  for (const [key, value] of Object.entries(vars)) {
    content = content.replaceAll(`{{${key}}}`, value);
  }
  return content.trim();
}
10 — Complete Data Flow

End-to-end: what happens when a user types /kata and picks "Execute T01".

1. User Action
User types /kata. The command handler calls showSmartEntry().
2. State Derivation
deriveState(basePath) reads .kata/ files: roadmap checkboxes, plan checkboxes, task summaries. Returns: active milestone, slice, task, and phase.
3. Wizard Menu
showNextAction() renders a TUI menu: "Execute T01", "Go auto", "View status". User picks "Execute T01".
4. Prompt Assembly
loadPrompt("guided-execute-task", { milestoneId: "M001", sliceId: "S01", taskId: "T01", taskTitle: "..." }) fills in the template.
5. Workflow Dispatch
dispatchWorkflow() reads KATA-WORKFLOW.md, concatenates ## Your Task\n\n{prompt}, sends as hidden message with triggerTurn: true.
6. before_agent_start Hook
Pi fires the hook. Kata's handler injects system prompt (contract + methodology + preferences). Detects the "Execute T01" prompt, builds GuidedExecuteContext with inlined task plan, carry-forward summaries, and resume state.
7. LLM Receives
The LLM sees three layers of context:
System Prompt
Pi base + Kata contract + methodology + preferences
Guided Context
Task plan, slice excerpt, prior summaries, resume state
Dispatch Message
Full protocol + "Execute T01" prompt
8. Agent Executes
The LLM follows the task plan: reads files, makes changes, runs tests, verifies must-haves, writes summary. All using pi's tools (read, edit, bash, etc.).
9. agent_end Hook
Turn completes. If auto-mode: save activity log, derive new state, determine next unit, create fresh session, dispatch next prompt. If manual: done — user sees the result.
The system's core insight: The LLM has no persistent memory between sessions. Everything it needs to continue working comes from (1) the methodology protocol injected into the system prompt, (2) structured prompt templates that tell it exactly what to do, and (3) .kata/ files on disk that it reads and writes as its external memory. The harness is just the plumbing that wires these three things together before each turn.
