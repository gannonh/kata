Kata Run, Step & Auto

user journey flows · i/o data pipelines · prompt architecture

/kata-run injects the full workflow protocol into the LLM and lets it self-direct. /kata step executes a single unit of work (plan, execute, verify, summarize). /kata auto wraps that in a supervised loop with crash recovery, budget control, stuck detection, and PR lifecycle management. Step and auto share the same backend, state derivation, and prompt template system. Run mode bypasses all of that.

1 — Overview

Run vs Step vs Auto at a Glance

Run Mode

Protocol injection, LLM self-directs

User invokes /kata-run. The command reads KATA-WORKFLOW.md and sends it to the LLM as instructions. The LLM reads state files itself, figures out the phase, and executes accordingly. No backend factory, no programmatic state derivation, no prompt templates.

Protocol-driven LLM self-directed No backend
Step Mode

Single unit, no loop

User invokes /kata step. The system derives state, picks the appropriate phase, builds a prompt, and dispatches it to the LLM. The LLM completes one unit. User reviews output, then manually invokes the next step.

Stateless per call User-paced Full visibility
Auto Mode

Supervised dispatch loop

User invokes /kata auto. The system bootstraps, derives initial state, then enters a loop: dispatch unit, wait for completion, derive next state, dispatch next. Includes timeout supervision, budget ceiling, crash recovery, stuck detection, and PR lifecycle management.

Loop control Crash recovery Budget guard PR lifecycle

What the LLM Receives

Message payload
Protocol
The full KATA-WORKFLOW.md from ~/.kata-cli/agent/ (~880 lines): hierarchy, file formats, phases, continue-here protocol, git strategy, summary injection rules, auto-mode contract. The LLM uses this as its operating manual.
Instructions
User Note
Optional free-text appended as ## User Note. Lets the user steer focus: "focus on error handling", "skip research", "start with S03".
Guidance

How Run Mode Differs from Step/Auto

Aspect /kata-run /kata step / /kata auto
State derivation LLM reads .kata/state.md or calls kata_derive_state itself Backend computes state programmatically
Phase routing LLM follows protocol instructions to determine phase Code routes to the correct prompt template by phase
Prompt construction Raw protocol document, no templates or variable injection Template loaded, variables injected, context inlined
Backend dependency None. No FileBackend or LinearBackend involved Requires backend factory and backend instance
Scope per invocation Open-ended. LLM decides what and how much to do One unit (step) or supervised loop of units (auto)
Visibility Message hidden (display: false), LLM output visible Prompt injected per unit, output visible per unit
When to use run mode: Best for sessions where you want the LLM to have the full methodology in context and self-direct its work, including reading state, deciding phases, and executing across multiple phases within a single conversation. Step and auto mode handle this programmatically, constraining the LLM to one unit at a time.

Key detail: Step mode is stateless per invocation. Each /kata step independently derives state from disk, so the user can freely edit files between steps without breaking the flow.

7 — Prompt Templates

Template Architecture

All prompts live in src/resources/extensions/kata/prompts/. The prompt loader reads {{variableName}} placeholders and substitutes them with values assembled by the backend. A system prompt (system.md) provides workflow rules and conventions for every dispatch.

9 — Auto-Mode Supervision

Safety & Durability Mechanisms

Crash Recovery

A lock file is written before each unit dispatch containing unitType, unitId, session file path, and completed unit count. On auto-mode startup, if a lock exists, the system reads the previous session's activity log and synthesizes a recovery prompt to resume from where it left off.

Lock file Session replay
Stuck Detection

Each dispatch compares the current unitType+unitId to the previous dispatch. If they match, a retry counter increments. After MAX_RETRIES (1), the system diagnoses the expected artifact that should have been written and stops auto-mode with a diagnostic message.

Unit comparison Retry limit
Budget Control

A metrics ledger persists cost and token usage per unit to disk. Before each dispatch, the accumulated cost is checked against the budget ceiling from preferences. When exceeded, auto-mode pauses with a cost report.

Metrics ledger Ceiling check
Timeout Watchdog

Three layers: a soft timeout warning at the configured soft_timeout_minutes, an idle watchdog that checks every 15 seconds for progress, and a hard timeout that forcefully stops auto-mode. For execute-task units, durability is checked before stopping (summary exists? task marked complete?).

Soft timeout Idle watchdog Hard timeout

Activity Logging

JSONL format. Each unit's activity is logged to .kata/activity/NNN-unitType-scope.jsonl. Logs include tool calls and outputs, enabling session replay for crash recovery and retry diagnostics. Sequential numbering (001, 002, ...) preserves execution order.
