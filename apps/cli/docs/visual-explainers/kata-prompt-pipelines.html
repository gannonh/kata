Kata Prompt Pipelines

How prompts flow from user command to LLM context across the four execution modes

Shared Foundation — Every Session
before_agent_start hook · index.ts

Fires once at session start for any project with a .kata/ directory. Builds the system prompt by concatenating these blocks in order:

1. System Prompt system.md
Agent identity, hard rules, skills table, commands, execution heuristics, playbooks
→
2. Workflow Mode "mode: linear"
Appended when entrypoint guard has a notice
→
3. Preferences Skill preferences block from .kata/preferences.md, resolved skill paths, warnings
→
4. New Skills XML block for skills installed during current auto-mode session (if any)
→
5. Workflow Protocol KATA-WORKFLOW.md
Full ~550-line workflow doc: hierarchy, artifacts, phases, formats, git strategy, auto-mode contract

Result: event.systemPrompt + "\n\n[SYSTEM CONTEXT — Kata]\n\n" + system.md + workflowMode + preferences + newSkills + KATA-WORKFLOW.md

Key constraint

KATA-WORKFLOW.md is always in the system prompt. No execution mode needs to re-send it. Auto-mode prompts reference it via "Consult KATA-WORKFLOW.md (injected into your system prompt)". Guided flow prompts say "Follow the Kata Workflow protocol in your system prompt."

Auto Mode — /kata auto
Entry · commands.ts → auto.ts

User runs /kata auto. The dispatch loop runs fresh sessions until the milestone is complete.

Step 1 — State derivation

backend.deriveState() → calls deriveLinearState() via Linear API. Returns KataState with phase, activeMilestone, activeSlice, activeTask.

Step 2 — Dispatch routing

resolveDispatchOptions() checks for UAT, reassessment, and research overrides. deriveUnitType(state, options) maps the phase to a unit type:

PhaseUnit TypeAuto-mode Prompt
pre-planningplan-milestoneplan-milestone.md
planningplan-sliceplan-slice.md
executing / verifyingexecute-taskexecute-task.md
summarizingcomplete-slicecomplete-slice.md
completing-milestonecomplete-milestonecomplete-milestone.md
replanning-slicereplan-slicereplan-slice.md

Override routes: research-milestone.md research-slice.md reassess-roadmap.md run-uat.md

Step 3 — Prompt building

backend.buildPrompt(phase, state, options) on LinearBackend builds a self-contained prompt. Each prompt template gets {{variable}} substitution with:

  • Inlined context: roadmap, slice plan, task plan, summaries — preloaded from Linear documents and issue descriptions
  • Backend rules: "CRITICAL: Do NOT create local .kata/ files. Use kata_write_document..."
  • Reference line: "Consult KATA-WORKFLOW.md (injected into your system prompt)"
  • Must-complete instruction specific to the unit type

Auto-mode prompts do advance issue state — they include kata_update_issue_state instructions.

Step 4 — Dispatch

pi.sendMessage({ customType: "kata-auto", content: finalPrompt }) with { triggerTurn: true }. The prompt may be prepended with crash recovery context or retry diagnostics.

Supervisors (soft timeout, hard timeout, idle watchdog) run in parallel. On session end, activity log is saved, issue state is advanced, and the loop restarts with fresh context.

Step Mode — /kata step (or bare /kata)
Entry · commands.ts

User runs /kata step or /kata (with an active milestone). Executes one unit in the current session — no loop, no fresh context.

Pipeline — same as auto, single pass
  1. createBackend(cwd) → backend.deriveState()
  2. If phase === "blocked" → notify and stop
  3. If phase === "complete" or no active milestone → fall through to showSmartEntry()
  4. Ensure slice branch (ensureSliceBranch) for slice-scoped phases
  5. backend.buildPrompt(phase, state) — same prompt templates as auto-mode
  6. Apply model preference via resolveModelForUnit(unitType)
  7. pi.sendMessage({ customType: "kata-step", content: prompt })

No dispatch options (no UAT/reassess/research overrides). No supervisors. No crash recovery. No retry loop. Just one shot.

Prompts loaded

Identical to auto-mode — LinearBackend.buildPrompt() selects the same template based on phase. The only difference is the delivery mechanism (kata-step customType, no loop).

Plan Mode — /kata plan
Entry · commands.ts → guided-flow.ts showPlan()

User runs /kata plan. Shows a state-dependent option picker, then dispatches via dispatchWorkflow().

State machine — 5 states
StateConditionOptions Presented
ANo milestonesPlan new milestone
BActive milestone, no roadmapPlan milestone roadmap, Discuss planning
CRoadmap + pending slicesPlan next unplanned slice, Pick a slice, Add slice, Resequence, Revise roadmap, Plan new milestone, Discuss planning
DAll slices completePlan new milestone, Add slices, Revise roadmap, Discuss planning
EAll milestones completePlan new milestone, Discuss planning
Prompt dispatch

dispatchWorkflow(ctx, pi, note) sends a user message:

"Follow the Kata Workflow protocol in your system prompt.\n\n## Your Task\n\n" + guided prompt content

The guided prompt is selected by the user's choice:

User ChoiceGuided PromptSource
Plan new milestoneguided-discuss-milestone.mdvia backend.buildDiscussPrompt()
Plan milestone roadmapguided-plan-milestone.mdvia loadPrompt()
Plan next/pick sliceguided-plan-slice.mdvia loadPrompt()
Add a new sliceguided-add-slice.mdvia loadPrompt()
Resequence slicesguided-resequence-slices.mdvia loadPrompt()
Revise roadmapguided-revise-roadmap.mdvia loadPrompt()
Discuss planningguided-discuss-planning.mdvia loadPrompt() with {{currentState}}
Critical difference from auto

All guided plan prompts contain: "Do NOT call kata_update_issue_state." Planning mode creates artifacts only — it does not advance execution state. Auto-mode plan prompts (same phase, different template) do advance state.

Discuss Mode — /kata discuss
Entry · commands.ts → guided-flow.ts showDiscuss()

User runs /kata discuss. Shows a slice picker (from the active milestone's roadmap), then dispatches a focused context interview for that slice.

Context preloading

buildDiscussSlicePrompt() reads from Linear and inlines:

  • Milestone roadmap (M001-ROADMAP)
  • Milestone context (M001-CONTEXT)
  • Milestone research (M001-RESEARCH)
  • Decisions register (DECISIONS)
  • Summaries from completed dependency slices

All context is inlined into the prompt so the agent starts with full awareness and doesn't need to re-read documents.

Prompt dispatch

Via dispatchWorkflow() → same mechanism as Plan mode:

"Follow the Kata Workflow protocol in your system prompt.\n\n## Your Task\n\n" + guided-discuss-slice.md + inlined context + output instructions

Output instructions tell the agent to write S01-CONTEXT via kata_write_document scoped to the slice issue.

Loop behavior

After the agent finishes the discussion, showDiscuss() loops back to the slice picker — the user can discuss another slice or exit. Each discuss dispatch waits for the agent to go idle via ctx.waitForIdle().
