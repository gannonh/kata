# Questioning

Project and milestone initialization are collaborative discovery, not form filling.

## Philosophy

Act as a thinking partner. The user may start with a fuzzy idea; help sharpen it by following the thread, making vague terms concrete, and surfacing assumptions.

Do not interrogate. Do not walk a checklist out loud. Ask one useful question at a time.

## Goal

By the end of questioning, you need enough clarity to write durable Kata artifacts that downstream skills can act on:

- What the user is building.
- Why it exists.
- Who it is for.
- What done looks like in observable terms.
- Constraints, non-goals, risks, and key decisions that affect planning.

## How To Question

- Start open: "What do you want to build?" or "What do you want to build next?"
- Follow energy: dig into what the user emphasizes.
- Challenge vagueness: "simple", "fast", "users", and "done" need concrete meanings.
- Make abstract ideas concrete: ask for a walkthrough or example.
- Clarify ambiguity with 2-4 concrete options when helpful.
- Stop when you can write the artifact; do not keep questioning for its own sake.

## Freeform Rule

If the user wants to explain in their own words, ask a plain text follow-up and wait. Do not force a multiple-choice interaction.

## Decision Gate

When you can write the durable artifact, ask:

```text
I think I understand what you're after. Ready for me to create the Kata project artifact?
```

Use equivalent wording for milestones, requirements, or roadmap gates. If the user wants to keep exploring, ask what they want to add or probe the largest remaining gap.

## Anti-Patterns

- Asking about technical experience.
- Asking canned enterprise questions before understanding the idea.
- Accepting vague answers without examples.
- Rushing into backend writes before the user has confirmed the shape.
- Treating alignment as a separate standalone skill instead of part of the active workflow.
