Symphony Orchestrator
Lifecycle

polling loop · agent sessions · prompt injection · state transitions

Symphony is a headless orchestrator that polls a Linear project, dispatches issues to isolated agent sessions, manages multi-turn conversations, retries with exponential back-off, and reconciles issue state each cycle. This document maps the full lifecycle — from poll tick to prompt delivery to completion.

3 — Worker Task Lifecycle

Each dispatched issue runs as an isolated tokio::spawn task. The worker ensures a workspace, runs hooks, starts an agent session, and loops through turns until the issue reaches a terminal state or max turns is hit.

1. Resolve Prompt
Call resolve_prompt_for_state() — selects per-state prompt file or falls back to monolith template. Inject {{ shared_context }} placeholder.
2. Ensure Workspace
Clone, worktree, or reuse existing workspace directory. Create branch <prefix>/<identifier>. Docker mode: start container, bootstrap repo inside it.
3. Run Hooks
Execute hooks.after_create (first workspace use), then hooks.before_run (every attempt). Hook failure on before_run is fatal — the worker exits.
4. Render Prompt
Run Liquid template engine: inject {{ issue.* }}, {{ attempt }}, {{ workspace.base_branch }}, {{ shared_context }}.
5. Start Session
Launch agent subprocess — codex app-server or kata --mode rpc. SSH remote: executes via ssh -T host bash -lc '...'.
6. Turn Loop
Turn 1 sends the rendered prompt. Turns 2+ send a continuation prompt. Between each turn: re-check issue state from Linear. Exits when: terminal state, different active state, max turns, or unassigned.
7. After-Run Hook
Execute hooks.after_run (non-fatal on failure). Stop agent session. Report result to orchestrator.
8. Completion
Orchestrator receives WorkerResult. Routes to continuation retry, failure retry, or completed. If cleanup_on_done: remove workspace directory.
4 — Prompt Assembly

Per-state prompts are built by concatenating up to four files, separated by ---. The system and repo preambles are injected every turn. The state-specific file changes based on the issue's Linear state at dispatch.

System
prompts/system.md
Agent identity, issue context template ({{ issue.identifier }}, {{ issue.title }}, {{ issue.description }}), retry/continuation awareness, Liquid variables. Repo-agnostic — same across all projects.
— — —
Repo
prompts/repo-sym.md (or repo-cli.md)
Build commands, test commands, directory layout, base branch, AGENTS.md reference. Repo-specific — varies per project.
— — —
Shared (legacy)
prompts/shared.md (optional, backward compat)
Single-file preamble from before the system/repo split. Still honoured if configured — slots between repo and state.
— — —
State-Specific
Resolved from prompts.by_state or prompts.default
The workflow instructions for this Linear state. In Progress → implement, test, push, open PR. Agent Review → address PR feedback. Merging → land the PR. Rework → close PR, fresh start.
Liquid template engine
Rendered Prompt (Turn 1)

All {{ issue.* }} and {{ workspace.base_branch }} variables are resolved. The {{ shared_context }} placeholder is replaced with cross-session context entries. This becomes the first user message to the agent.

State key normalization. State names from Linear (e.g. "In Progress") are lowercased and trimmed before matching against prompts.by_state keys. If no match and no prompts.default, falls back to the monolith template body from WORKFLOW.md.
5 — Multi-Turn Sessions

A single worker attempt can run multiple turns within one agent session. Turn 1 receives the full rendered prompt. Turns 2+ receive a short continuation prompt. Between turns, the orchestrator checks if the issue is still in an active state.

Turn 1 Prompt

Full concatenated prompt: system + repo + state-specific. Contains issue description, labels, URL, attempt number, workspace context.

Turn 2+ Prompt

Short continuation: "This is continuation turn #N of M. Resume from current workspace state. Focus on remaining ticket work." The agent still has the full Turn 1 context in its conversation history — the session subprocess persists across turns.

Stall detection. If no agent events are received within stall_timeout_ms (default 5 min), the session is force-terminated and the issue is retried with exponential backoff.
6 — Retry & Continuation

After a worker completes, the orchestrator decides the next step based on the outcome and issue state. Continuations re-dispatch quickly. Failures use exponential backoff up to max_retry_backoff_ms.

Continuation

Issue still active after max turns. Re-dispatched quickly with the same workspace. Attempt counter does not increment. Fresh prompt is re-resolved for the current state.

Failure Retry

Session crashed, stalled, or errored. Exponential backoff: attempt × base delay, capped at max_retry_backoff_ms (default 5 min).

7 — Issue State Machine

Tracker issue states drive the lifecycle. Active states make issues eligible for dispatch. Terminal states signal completion and trigger workspace cleanup. The agent itself transitions issues between states via the prompt instructions.

Each state maps to a different prompt file, giving the agent state-specific instructions.

Terminal States (session ends)

Done Closed Cancelled Duplicate

Worker stops. No continuation scheduled. Workspace cleaned up if cleanup_on_done: true.

Prompt-per-state. The same issue can cycle through multiple states and get different prompts each time. A Todo issue gets the in-progress.md prompt. After it pushes a PR and moves to Agent Review, the next dispatch (continuation retry) resolves agent-review.md — giving the agent entirely different instructions for addressing review feedback.
