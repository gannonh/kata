Three Approaches for Intent Capture & Spec Elicitation

How the orchestrator turns user intent into a rich spec

A

Skill-Only: Pure Prompt Engineering

A single system skill (SKILL.md) contains the entire elicitation flow: phase definitions, checkpoint list, spec template guidance, and examples. The system prompt gets a small addition that detects new-project lifecycle and auto-invokes the skill.

Pros

  • Minimal code changes — mostly prompt/skill authoring
  • Easy to iterate on elicitation behavior by editing markdown
  • Ships fast, validates the concept
  • Same pattern as superpowers brainstorming skill

Cons

  • No spec state tracking — agent relies on conversation context alone
  • No persistence — spec only exists inline in chat
  • Lifecycle detection is fragile (prompt-based heuristic)
  • Can't resume elicitation across sessions
B

Skill + Lifecycle Signal + File Persistence

Same system skill approach, but with two code-level additions: (1) inject session lifecycle context into user messages so the orchestrator knows it's a new project, and (2) persist the finished spec as a markdown file in the workspace directory.

Pros

  • Reliable lifecycle detection via injected context (not heuristic)
  • Spec persists as a file — ready for KAT-273 spec panel
  • Skill still drives the elicitation behavior (easy to iterate)
  • Moderate code changes, focused and contained

Cons

  • More moving parts than pure skill approach
  • Need to define spec file format and storage location
  • Spec state tracking still limited (file exists or doesn't)
C

Skill + Lifecycle + State Machine + Persistence

Full infrastructure: system skill for elicitation, lifecycle context injection, spec file persistence, AND a spec state model (gathering → draft → approved) tracked on session metadata alongside todoState.

Pros

  • Complete foundation for the spec-driven workflow
  • Orchestrator can resume elicitation where it left off
  • State visible in UI (center panel status indicators)
  • Clean handoff to KAT-273 and KAT-275

Cons

  • Largest scope — touches session metadata, event processor, UI
  • Risk of over-building before validating the elicitation flow itself
  • State machine design may change once spec panel (KAT-273) lands

Recommendation: Approach B

B hits the right balance. The skill drives the elicitation (easy to iterate), the lifecycle signal makes detection reliable (not a prompt hack), and file persistence means the spec survives as an artifact for follow-on epics. It avoids building a state machine that may change shape once KAT-273 lands.

The elicitation skill itself is the high-value deliverable. Lifecycle injection and file persistence are small, focused code changes that make the skill work reliably.
